# Getting Support

This document explains where to get help with StellarLens. Please use the channel that matches your need so that the maintainers can respond efficiently and so that other users can find the answers later.

## Questions and Discussion

For general questions, design discussion, usage help, and "is this the right approach?" conversations, use [GitHub Discussions](https://github.com/ezedike-evan/stellar-lens/discussions). Discussions are searchable and benefit other users with the same question.

## Bug Reports

If you have found a reproducible bug in the SDK, open a [GitHub Issue](https://github.com/ezedike-evan/stellar-lens/issues/new/choose) using the **Bug Report** template. Please fill the template out completely — the requested version, environment, and reproduction details are required to triage the report.

## Feature Requests

To propose a new feature or an enhancement to an existing one, open a [GitHub Issue](https://github.com/ezedike-evan/stellar-lens/issues/new/choose) using the **Feature Request** template. Describe the use case you want supported and, if possible, the API shape you have in mind.

## Security Vulnerabilities

**Do not open a public issue, discussion, or pull request for a security vulnerability.** Follow the process described in [SECURITY.md](./SECURITY.md), which routes reports to a private channel and defines disclosure timelines.

## Documentation

User-facing documentation is published at the project documentation site:

```
https://stellar-lens.dev/docs
```

If you find an error in the documentation or a section that is unclear, please open a bug report or, for small fixes, a pull request directly.

## Response Times

StellarLens is currently maintained on a best-effort basis by a small team. Critical bugs and security issues are prioritised; response times for non-critical issues, feature requests, and discussion threads may vary depending on maintainer availability. Pull requests that include tests and follow the contribution guidelines in [CONTRIBUTING.md](./CONTRIBUTING.md) are the fastest route to getting a change shipped.
